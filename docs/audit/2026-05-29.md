# Security & Hardening Audit — 2026-05-29 (v2.4.6)

P(-1) hardening step closing out the **2.4.x collision-correctness arc**. Scope:
memory-safety / allocation-overflow review of the whole library, supply-chain
integrity, dependency CVE posture (known + web research), and a security review
of the six collision algorithms brought into the build chain across 2.4.0–2.4.5.

**Verdict: posture is solid.** No new vulnerability. The three genuine bugs in
the arc were already fixed (hull sort + undefined `f64_le`/`f64_ge` in 2.4.0; MPR
origin-containment in 2.4.4; impulse-sign/convergence in 2.4.5). The one open
item — stdlib `mat_new` integer overflow — is upstream and mitigated in hisab's
usage. Six allocation-guard regression tests added; suite 895 → 901.

## Methodology

- **Internal**: grepped the full attack surface — shell/exec/FFI/libc, every
  `alloc(<expr>)`, every iterative loop bound, raw `load64`/`store64` indexing.
- **External (CVE)**: searched CVE/advisory sources for the Cyrius/cycc toolchain
  and for the CWE-190 allocation-overflow class.
- **Supply chain**: `cyrius deps --verify` against `cyrius.lock`, `cyrius vet`.

## Findings

### 1. No command-injection / FFI / syscall surface — PASS
No `sys_system`, `exec`, `popen`, `fork`, FFI, or libc anywhere in `src/`. The
library issues no syscalls (the CLI smoke binary only `exit`s). Nothing for a
CWE-78 (OS command injection) or unsafe-FFI class to attach to.

### 2. Allocation overflow (CWE-190 → CWE-787) — PASS (mitigated)
The library uses raw `alloc()` + `load64`/`store64` with no bounds checks, so
under-allocation from an overflowing size computation is the primary memory-safety
class (cf. recent CVEs in CImg `W*H*D`, kissfft, glibc — see Sources).

- hisab's own dimension-taking allocators **all guard the multiply before
  allocating**: `tensor_new` (rank ≤ 8, `total > MAX/dim` check, `_TENSOR_MAX_ELEMS
  = 1048576`), `cmat_new` (`rows > 65536/cols` check), `christoffel`/`riemann`
  (dim ≤ 16), `num_sieve` (≤ 10M). Pinned by new regression tests (see below).
- The 106 single-multiply `alloc(n*8)` sites overflow only for `n > 2⁶¹`
  (≈ 2.3×10¹⁸); such `n` would exhaust memory in the allocator long before the
  product wraps. No two-variable multiply-alloc exists in the collision code
  (it uses auto-growing `vec`s and fixed-size structs).
- **Open (upstream)**: stdlib `lib/matrix.cyr` `mat_new(rows, cols)` computes
  `16 + rows*cols*8` with **no guard** (still present in the pinned 6.0.14
  snapshot). hisab calls it in `linalg_ext` (SVD/QR) and `optimize` (LM/Newton),
  but always with dimensions taken from an **already-allocated** input matrix —
  i.e. a product that already passed the allocator — never from a raw untrusted
  scalar. hisab's own raw-dimension constructor (`cmat_new`) is guarded. The fix
  belongs in cyrius stdlib (hisab must not edit vendored `lib/`); tracked as
  roadmap **2.5.0**. Threat-model note updated from the stale "cyrius 5.0.1".

### 3. Unbounded iteration / DoS — PASS
Every iterative path is capped: MPR refinement `_COL_MAX_ITER = 64`, ear-clipping
`n*n` safety bound, half-edge one-ring walk `< 1000` guard, all numeric solvers
`max_iter` + `ERR_NO_CONVERGENCE`. The 2.4.x additions introduced no uncapped loop.

### 4. Collision algorithms (2.4.x) memory safety — PASS
All index access goes through `vec_get`, which **traps** (controlled abort) on an
out-of-bounds index rather than corrupting memory — the 2.4.0 `convex_hull_2d`
bug surfaced exactly this way (`vec: index < 0`) instead of a silent OOB write.
The raw-pointer `tris` array in `delaunay_2d` is bounded by `n_tris`. No OOB write
path found.

### 5. Supply chain — PASS
`cyrius deps --verify` → **60 verified, 0 failed** against committed `cyrius.lock`.
`cyrius vet` → **2 deps, 0 untrusted, 0 missing**. No third-party dependencies:
only the cyrius stdlib and first-party **sakshi**. No FFI, no libc. The
third-party-CVE attack surface is therefore **zero**.

### 6. Dependency CVE posture — PASS
No CVEs or security advisories exist for the Cyrius language / cycc compiler — the
toolchain is a niche sovereign project not indexed in public CVE databases (NVD,
etc.). With no third-party runtime dependencies, hisab inherits no third-party
CVE exposure. Supply-chain integrity is enforced by the SHA-locked `cyrius.lock`.

## Changes

- **`tests/edge_cases.tcyr`**: 6 allocation-overflow-guard regression assertions
  (`cmat_new` huge/zero/valid; `tensor_new` rank>8 / overflow-shape / valid),
  pinning the CWE-190 mitigations.
- **`docs/development/threat-model.md`**: added the six 2.4.x collision algorithms
  to the attack-surface table; refreshed the `mat_new` note for 6.0.14 + roadmap
  2.5.0; recorded supply-chain verification and this audit in the history.

## Open items (not regressions)

- stdlib `mat_new` overflow guard — **upstream**, roadmap 2.5.0 (verify/fix in
  cyrius; hisab's usage is mitigated as above).
